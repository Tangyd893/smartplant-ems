# SmartPlant EMS 软件需求规格说明书

## 1. 引言

### 1.1 编写目的
本文档用于明确 SmartPlant EMS 的功能需求、非功能需求、接口规范和约束条件，作为后续设计、开发、测试的依据。

### 1.2 范围
本系统为智慧工厂能源管理系统（SmartPlant EMS），面向中型制造企业的能源管理场景，支持设备管理、能耗监控、告警管理、报表分析等核心功能。

### 1.3 定义与缩写

| 术语 | 定义 |
|------|------|
| EMS | Energy Management System，能源管理系统 |
| 设备 | 具有能耗采集能力的生产设备，如空压机、注塑机等 |
| 能耗记录 | 设备在某一时刻的功率、用电量等采集数据 |
| 告警 | 设备能耗异常时触发的通知信息 |
| 峰谷平 | 电价的三个时段：峰时段、谷时段、平时段 |

## 3. 功能需求

### 3.1 模块划分

系统划分为 4 个功能模块：

| 模块 | 描述 | 优先级 |
|------|------|--------|
| **设备管理** | 设备的增删改查、状态管理 | P0 |
| **能耗分析** | 能耗数据采集、统计、趋势分析 | P0 |
| **告警管理** | 告警触发、通知、处理 | P1 |
| **报表管理** | 报表生成、导出、查看 | P1 |
| **系统管理** | 用户登录、权限管理（预留） | P2 |

### 3.2 设备管理模块

#### 3.2.1 功能列表

| 功能 | 功能描述 | 接口路径 |
|------|---------|---------|
| 设备列表 | 分页查询设备，支持类型/状态筛选 | `GET /api/device/list` |
| 设备详情 | 获取单个设备详细信息 | `GET /api/device/:id` |
| 创建设备 | 添加新设备 | `POST /api/device` |
| 更新设备 | 修改设备信息 | `PUT /api/device/:id` |
| 删除设备 | 删除设备（软删除） | `DELETE /api/device/:id` |
| 设备实时数据 | 获取设备当前功率/电压等 | `GET /api/device/:id/realtime` |

#### 3.2.2 数据模型：设备

```go
type Device struct {
    ID           int64   // 设备ID（主键）
    DeviceCode   string  // 设备编号（唯一，如 "AC-001"）
    DeviceName   string  // 设备名称
    DeviceType   string  // 设备类型：air_compressor / injection_machine / laser
    Location     string  // 安装位置
    Status       int     // 状态：0=停用, 1=运行, 2=维护, 3=故障
    PowerRating  float64 // 额定功率(kW)
    CreatedAt    time.Time
    UpdatedAt    time.Time
}
```

#### 3.2.3 设备类型枚举

| 类型值 | 显示名称 | 说明 |
|--------|---------|------|
| `air_compressor` | 空压机 | 空气压缩设备 |
| `injection_machine` | 注塑机 | 塑料注塑设备 |
| `laser` | 激光切割机 | 金属切割设备 |

#### 3.2.4 状态枚举

| 值 | 状态 | Badge 颜色 |
|----|------|-----------|
| 0 | 停用 | 蓝灰 |
| 1 | 运行中 | 绿色 |
| 2 | 维护中 | 黄色 |
| 3 | 故障 | 红色 |

#### 3.2.5 用例

**UC-D001：查看设备列表**
1. 用户进入"设备管理"页面
2. 系统默认展示第 1 页，每页 20 条
3. 用户可按设备类型、状态筛选
4. 支持按设备编号/名称搜索

**UC-D002：查看设备详情**
1. 用户点击设备列表中的"详情"按钮
2. 系统展示设备基本信息 + 实时数据面板

**UC-D003：创建设备**
1. 用户点击"添加设备"按钮
2. 填写设备编号、名称、类型、位置、额定功率
3. 系统校验数据（编号唯一性、必填项）
4. 保存并返回成功

```go
}
```

## 4. 非功能需求

### 4.1 性能需求

| 指标 | 要求 |
|------|------|
| 页面加载时间 | 首屏 < 2s（网络正常情况下） |
| API 响应时间 | 平均 < 200ms，p99 < 500ms |
| 并发支持 | 支持 50 个并发用户 |
| 数据写入频率 | 设备数据采集频率 ≤ 1次/分钟 |

### 4.2 可用性需求

| 指标 | 要求 |
|------|------|
| 系统可用时间 | 24小时 × 7天 |
| 故障恢复时间 | MTTR < 2小时 |
| 数据备份 | 每日自动备份，保留 7 天 |

### 4.3 安全性需求

- 所有 API 需要认证（JWT Token）后才可访问
- 用户密码需要加密存储（bcrypt）
- 敏感配置信息不得明文写入代码（使用环境变量）
- SQL 注入防护：使用参数化查询

### 4.4 可维护性

- 代码注释覆盖率 > 60%
- 核心模块需要有单元测试
- API 文档需要保持更新

## 5. 接口规范

### 5.1 统一响应格式

```json
// 成功响应
{
  "code": 0,
  "msg": "success",
  "data": { ... }
}

// 分页响应
{
  "code": 0,
  "msg": "success",
  "data": {
    "records": [...],
    "total": 100,
    "page": 1,
    "size": 20
  }
}

// 错误响应
{
  "code": 400,
  "msg": "参数错误"
}
```

### 5.2 错误码定义

| 错误码 | 含义 |
|--------|------|
| 0 | 成功 |
| 400 | 请求参数错误 |
| 401 | 未认证（未登录/Token 失效） |
| 403 | 无权限 |
| 404 | 资源不存在 |
| 500 | 服务器内部错误 |

### 5.3 分页参数

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| page | int | 1 | 页码（从 1 开始） |
| size | int | 20 | 每页条数（最大 100） |
